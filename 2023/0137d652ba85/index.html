MessageQueue 简介

在 RokcetMQ 中有几个核心的概念，其中就包含 Topic 和 Queue

  • Topic 是业务上的逻辑分类，一类消息创发往同一个 Topic
  • Queue 则是消息实际发送的地方，一个 Topic 对应多个 Queue

MessageQueue 结构

由上面 MessageQueue 属性可知，想要确定消息到底发送到那个 Queue 中需要知道三个信息

  • topic: 这个不用多说，发送肯定需要先知道发往哪个 Topic
  • brokerName: 为什么还需要知道 BrokerName 呢？主要是某个 Topic 可以创建在集群中的多个 Broker 中，比如集群中有 5 个 Broker, 我们可以将 Topic 创建在任意几个或全部的 Broker 中
  • queueId：Broker 中创建的 Topic 默认是会有四个 Queue, 所以需要知道具体是发送到哪个 Queue 中

为什么不推荐在生产环境自动创建 Topic

  • 安全性问题：自动创建 Topic 可能会导致未经授权的 Topic 被创建，从而增加了系统的安全风险。
  • 风险控制问题：如果允许自动创建 Topic，可能会导致系统中出现大量不必要的 Topic，增加了管理和维护的难度。
  • 配置管理问题：手动创建 Topic 可以更好地控制 Topic 的配置，包括分区数、副本数等，而自动创建 Topic 则可能会导致配置的混乱和不一致。
  • 性能问题：大量的不必要的 Topic 会增加系统的负载和资源消耗，影响系统的性能表现

选择 MessageQueue 逻辑重点2 – Broker的避让

对于选择 MessageQueue 还需要关注的就是有一个选择 Broker 时的 避让 逻辑，即当前面发送消息失败时，在重试的时候不会选择到上一次发送的 Broker ( 因为 Queue 是分配在 Broker 上的，所以选择 Queue 也可以认为就是选择 Broker )

为什么需要避让逻辑而不是直接删除 Broker 信息

当选择一个 Broker 发送消息时若发送失败，说明 Broker 可能宕机了，对于此次消息发送而再次选择时最好时避免选到该 Broker，所以需要记录上一次发送的 Broker, 再次选择时避免选择到

那么为什么要记录上一次发送失败的 Broker，而不是直接删除失败的 Broker 信息呢？因为 Producer 会定期从 NameServer 获取 Broker 的状态，所以维护 Broker 状态的这件事情其实是有另外的线程在做，那么对于此次发送消息的线程来说只要简单排除一下发送失败的 Broker 即可

readQueue 和 writeQueue 的作用

在 RocketMQ 中 Queue 的数量分为 readQueueNums 和 writeQueueNums， 通过 org.apache.rocketmq.common.protocol.route.QueueData 的属性也能看出来

不过这里需要注意的是，这并不代表有两种类型的 Queue, 这里的 readQueueNums 和 writeQueueNums 只是单纯的代表数量

readQueueNums
readQueueNums 是对消费者起作用，因为消费者启动时需要读取 Queue 的数量（这个数量就是 readQueueNums ），然后将消费者与 Queue 进行绑定，也就是每个消费者需要消费哪些 Queue 的消息，物理上实际存在的 Queue 的数量

writeQueueNums
writeQueueNums 是对生产者起作用，因为生产者启动时需要知道 Queue 的数量（这个数量就是 writeQueueNums ），然后选择一个 Queue 将消息写入，物理上会创建的 Queue 的数量是等于 writeQueueNums 的。

  • 一般情况下 readQueueNums = writeQueueNums
  • 当 readQueueNums > writeQueueNums, 比如 readQueueNums=5， writeQueueNums=3 时，此时的场景如下图所示

readQueueNums > writeQueueNums 的场景

  • 这种场景是为了平滑扩容消费者的场景准备的， 消费者的并发量是不可能超过 Queue 的数量的，要想增加消费者，很可能就要增加 Queue, 假如只有一种类型的 Queue, 当增加 Queue 之后，原来每个消费者的压力可能会增大，这是有一定风险的
  • readQueue 和 writeQueue 分开之后，就可以在不停机的情况下先增加 readQueue 的数量，这些增加的 readQueue 是没有数据的，然后再增加消费者，这样消费者压力并没有增大，最后再增加 writeQueue 数量和 readQueue 数量一致，就完成了扩容

writeQueueNums > readQueueNums 的场景

  • 这种场景是为了缩容的场景准备的， 先减少 writeQueue 的数量， 比如从 5 -> 3, 当 4 号和 5 号 Queue 消费完成后，再将 readQueue 数量减少到 3， 最好停掉部分消费者，就完成了缩容

批量消息

  • 批量消息发送是将同一主题的多条消息一起打包发送到消息服务端，减少网络调用次数，提高网络传输效率
  • 单批次消息发送总长度不能超过 MQProducer#maxMessageSize
  • 批量消息发送要解决的是如何将这些消息编码以便服务端能够正确解码出每条消息的消息内容

非批量消息使用的消息承载结构是 Message 类
批量消息使用的消息承载结构是 MessageBatch

MessageBatch 继承了 Message, 同时含有 List<Message> messages 属性，所以消息发送和单挑消息发送逻辑没有任何区别（ java 多态的使用，方法中接收 Message 类型的参数也可以接收 MessageBatch ）

每一条消息传递的时候最终是要序列化成 byte[], 批量消息的发送流程没有任何变化，但是消息编码和解析是不同的，在 MessageBatch 中有一个核心方法就是对消息进行编码的

最终所有的消息内容存放在 byte[] 中，那么要怎么区分每一个消息占用的部分是多少呢？ 比如一共 80 字节，5个消息，要知道每个消息的内容不同，消息长度肯定不同，所以不可能是平分，所以必然是要设计出 不变的结构 + 可变的内容
