RocketMQ消息存储机制

RocketMQ版本： 4.9.7
JDK版本： 17

存储相关基础知识

  • RocketMQ 主要存储的文件包括 Comitlog 文件、ConsumeQueue文件、IndexFile文件
  • RocketMQ 将所有主题的消息存储在同一个文件中，即一个 Commitlog 文件，这么做的优势是确保消息发送时顺序写文件，提高消息发送的性能和吞吐量, Commitlog 文件存储目录默认为 ${ROCKET_HOME}/store/commitlog 目录下
  • ConsumerQueue 则是为了提高消费者的消费能力而设计的一个文件，Commitlog 包含所有 Topic 的消息，但是消费者是基于某个 Topic 的订阅机制，所以就增加了 ConsumerQueue 消息队列文件，消息存储在 Commitlog 文件后，将异步转发到消息消费队列
  • IndexFile 索引文件则是为了加速消息的检索性能，根据消息的属性快速从 Commitlog 文件夹中检索消息
  • ${ROCKET_HOME}/store/config 目录主要存储运行期间的配置信息
    • consumerFilter.json：主题消息过滤信息
    • consumerOffset.json：集群消费模式消息消费进度
    • delayOffset.json：延时消息队列拉取进度
    • subscriptionGroup.json：消息消费组配置信息
    • topics.json: topic 配置属性
  • ${ROCKET_HOME}/store/abort ： 如果存在 abort 文件说明 Broker 非正常关闭，该文件在默认启动时创建，正常退出之前删除
  • ${ROCKET_HOME}/store/checkpoint: 文件检测点，存储commitlog文件最后一次刷盘时间戳、consumequeue最后一次刷盘时间、index索引文件最后一次刷盘时间戳

ConsumerQueue 文件

  • Commitlog 文件是此时的所有消息的内容，ConsumerQueue 文件可以认为是按照 Topic 对 Commitlog 进行分类和索引(因为 ConsumerQueue 并不会存储完整的消息内容，只是会存储消息在 Commitlog 文件中的偏移量，想要获取完整的消息还是需要到 Commitlog文件中获取)

  • ConsumerQueue 文件存储的内容也是结构性的，该文件中的每一条数据称为 ConsumerQueue 条目，每一个 ConsumerQueue 条目的结构如下
  • 单个 ConsumerQueue 文件默认包含 30万个条目，单个文件的大小就是 30w * 20字节
  • 因为 ConsumerQueue 条目大小是固定的(8+4+8 字节)，所以可以将 ConsumerQueue 文件看成是 ConsumerQueue 条目的数组，下标就是 ConsumerQueue 的逻辑偏移量，消息消费进度存储的偏移量也是逻辑偏移量
  • ConsumerQueue 文件新增的时机是当消息到达 Commitlog 文件后，会有专门发线程产生消息转发任务，此时会添加数据到 ConsumerQueue 中

索引文件结构

接下来就看一下索引文件结果，下图来自 《RocketMQ技术内幕:RocketMQ架构设计与实现原理》 一书

整体看 Index 索引文件包含三个部分

  • head： 文件头
  • slot： 哈希槽
  • index content： 索引内容

文件头(40字节)

文件头一共包含 40 字节，从前往后代表的含义如下

  1. 开始时间( beginTimestamp ), 占用 8 字节
  2. 结束时间( endTimestamp ), 占用 8 字节
  3. 消息最小物理偏移量( beginPhyoffset ), 占用 8 字节，最小物理偏移量在往索引文件添加第一个条目的时候就可以确定，因为消息是顺序添加的，第一个添加的消息就是物理偏移量最小的
  4. 消息最大物理偏移量( endPhyoffset ), 占用 8 字节，最大物理偏移量是随着往索引文件不断添加条目而递增的
  5. 有效 hash slot count, 占用 4 字节，最坏的情况下有效的哈希槽只有一个（所有消息的 Keys 字段都相同，导致 key 的 hash 冲突），最理想的情况下每个 索引文件可以包含 500w 个 hash slot
  6. index 索引数量，占用 4 字节，默认一个索引文件可以包含 2000万个 index content, index count 就是记录的已经使用的 index content 数量

slot

  • slot 区域默认可以存放 500万个，可以认为是长度为 500w 的数组
  • slot 中存放的是当前 index content 的位置，这个位置是逻辑偏移量，为什么可以记录逻辑偏移量呢？先给出计算公式再进行说明

上面公式可以成立的原因就是 头部长度固定， slot区域总长度也固定，每一个index content长度固定，所以就可以根据索引下标来计算实际位置

如何根据key确定slot位置
  1. 根据 Message 中的 Keys 计算 hashcode
  2. 使用 hashcode 对 slot 总数取模
keys hash 冲突如何解决

在如下两种情况下都可能存在 keys 的哈希冲突

  1. 不同消息的 Keys 字段相同，此时计算出来的 hashcode 相同
  2. 根据 keys 计算出来的 hashcode 不同，但是对 slot 长度取模后得出的结果相同

当出现这种冲突时，slot 中存放的是最后一次 index content 的位置？也就是说存放的之前的 index content 位置被覆盖了，为什么可以这样呢？这是因为 index content 中会有字段来存储之前的 index content 的位置

index content (一共20字节)

  • hashcode 值： 4字节
  • 消息的物理地址(phyoffset): 8字节，消息的物理地址指向的是 Commitlog 文件中的偏移量
  • 当前消息与最早消息的时间差（timedif）: 4字节，这个是为了时间的范围查询
  • 上一条 index content 的位置（pre index no）:4字节，前面讲 slot 冲突时，slot 中记录的是最后一个 index content 的位置，然后再由这个 index content 指向之前那个 index conent

下面以两次往 Index 索引文件添加内容为例，看一下当 key 相同时是如何处理冲突的
第一次
假设根据 keys 计算出来的 slot 下标是5，因为是第一次添加，所以 index content 就放在第一个位置(index content 和 slot 不同，slot 需要计算下标位置，index content 直接依次递增存放)，此时结构图如下所示

第二次
第二个发送 Message 时还是使用相同的 keys, 那么计算出来的 slot 位置肯定还是 5

首先，slot 区域中下标为5的位置存储的内容改变了，变成当前新增的 index content 的逻辑偏移量(即 index content 区域的下标)
其次，当前 index content 中的 pre index no 区域中会存储之前的 index content 的逻辑偏移量(因为有哈希冲突了，如果没有冲突就不用存储)

MappedFile 和 MappedFileQueue

  • Commitlog 文件存储目录为 ${ROCKET_HOME}/store/commitlog，每一个文件默认都是 1G, 如下图所示
  • Commitlog 文件名就是文件中第一条消息的物理量偏移量
  • MappedFile 就是对应这一个个的 Commitlog 文件，是内存映射文件的具体实现
  • MappedFileQueue 是 MappedFile的管理容器，对应 ${ROCKET_HOME}/store/commitlog 文件夹
