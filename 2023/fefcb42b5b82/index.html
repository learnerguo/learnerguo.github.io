Mysql 中锁的概念

锁是数据库系统区别于文件系统的一个关键特性。锁机制用于管理对共享资源的并发访问

Mysql 中与锁相关的表

Mysql5.* 版本

  1. information_schema.INNODB_TRX
  2. information_schema.INNODB_LOCKS
  3. information_schema.INNODB_LOCK_WAITS

Mysql8 版本

  1. information_schema.INNODB_TRX
  2. performance_schema.INNODB_LOCKS
  3. performance_schema.INNODB_LOCK_WAITS

共享锁和互斥锁

共享锁简写为 S
意向共享锁简写为 IS
排他锁简写为 X
意向排他锁简写为 IX

锁类型独占锁共享锁
独占锁互斥互斥
共享锁互斥不互斥
  • Mysql 查询操作默认默认不会加锁
  • Mysql 在读和写之间不会互斥，因为对于读操作是直接使用 MVCC 来实现
  • 手动给 select 增加共享锁的方式是 select * from table_name lock in share mode;
  • 手动给 select 增加独占锁的方式是 select * from table_name for update;

表级锁

表级锁分为两种，分别是 表锁 和 表级意向锁

表锁

  1. 表级共享锁： LOCK TABLES xxx READ
  2. 表级独占锁： LOCK TABLES xxx WRITE

可以使用 unlock tables; 来释放所有的表级锁

表级意向锁

  • 更新数据时，除了会增加行级锁外，还会增加一个 表级意向独占锁
  • 在一个事务中，如果执行了查询操作，那么会在对应的表级加一个意向共享锁

两种表级锁的互斥关系

手动添加的表锁和表级意向锁是存在互斥关系的，对应的互斥关系如下

锁类型(都是表级)独占锁意向独占锁共享锁意向共享锁
独占锁互斥互斥互斥互斥
意向独占锁互斥不互斥互斥不互斥
共享锁互斥互斥不互斥不互斥
意向共享锁互斥不互斥不互斥不互斥

表级意向锁的作用

  • InnoDB 存储引擎对标的行记录添加 S 锁之前，会先在表级别添加一个 IS 锁
  • InnoDB 存储引擎对标的行记录添加 X 锁之前，会先在表级别添加一个 IX 锁
  • IS 锁和 IX 锁的作用是为了后续在加表级别的 S 锁和 X 锁时判断表中是否有已经被加锁的记录，以避免用遍历的方式来查看表中有没有上锁的记录，下面举例说明

事务 A 修改一条数据，所以事务 A 会给这条数据增加 行锁(下面会讲解)

事务 B 想要添加表级排他锁，因为是表级，所以可以理解成会给所有行都上锁，因为事务 A 已经上锁成功，事务 B 肯定是无法加锁成功的，这点没有疑问，问题就是判断的过程，事务 B 怎么知道哪一行加锁了呢? 那只能一行一行去判断了，这样效率就很低了，所以才有了意向锁，也就是说实际情况会是下面这样的

事务 A 修改一条数据，所以事务 A 会给这条数据增加 行锁(下面会讲解)，同时会给这张表加一个意向排他锁

事务 B 想要添加表级排他锁，一看发现这个表上面已经存在了 意向排他锁，根据上面的互斥关系表，可以知道，表级排他锁和表级意向排他锁是互斥的，那事务 B 就不能加锁成功了，这样只需要判断一次，效率明显提高

行级锁

  • 前面讲解的是表级锁，表级意向锁是 InnoDB 自动帮我们添加的，修改数据时会自动帮我们添加 表级意向独占锁，至于手动添加的情况比较少，接下来需要讲解的就是行级锁了
  • 行级锁锁住的是索引

行级锁的分类

  • Record Locks： 记录锁， 官方正式的名称是 LOCK_REC_NOT_GAP，对对应的索引记录项加锁， 表中展示的是 X,REC_NOT_GAP
  • Gap Locks： 间隙锁，锁住的是索引之间的间隙，** 表中展示的是 X,GAP**
  • Next-key Lock： 邻键锁 Next-key Lock = Gap Locks + Record Locks, 表中展示的是 X
  • Insert Intention Locks： 插入数据时需要插入的位置是否被别的事务添加了 Gap 锁（也就是 Gap Locks 或 Next-key Lock ），如果有的话，当前事务的插入操作需要等待，直到拥有 Gap 锁的事务提交。此时对于等待的那个事务就会添加一个 Insert Intention Locks
