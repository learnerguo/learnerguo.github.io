分布式架构原理与实践

发表于2024-01-06|更新于2024-01-08|java架构
|字数总计:1.5k|阅读时长:4分钟|阅读量:

参考书籍： 《分布式架构原理与实践》 崔皓

分布式整体框架内容

该书中谈论的分布式架构包含的内容

  • 分布式应用服务的拆分
  • 分布式调用
    • 接入层的负载均衡（终端到服务集群的负载，比如南方客户访问同一个业务可以路由到南方服务中心而不是北方服务中心，就近服务可以提高访问性能）
    • 内部服务的负载均衡，也就是API网关做的事情（即使路由到南方中心，但内部服务也是集群部署，要定位到某个具体的服务器还需要进一步负载）
    • 服务注册与发现
    • 服务间的远程调用：HTTP调用，RPC 调用等
  • 分布式协同
    • 分布式系统的特性与互斥问题
    • 分布式锁
    • 分布式事务
    • 分布式选举
  • 分布式计算
    • 现在都是大数据，单一服务比较难处理，所以会使用分布式计算，即集群中的每个服务计算部分数据
    • 批量静态数据计算的 MapReduce 模式
    • 动态数据流进行计算的 Stream 模式
  • 分布式存储
    • 分布式架构中对服务进行了拆分，每个服务都会产生数据，每个服务的数据是分开存储的
  • 分布式资源管理和调度
  • 高性能和高可用以及指标与监控

分布式调用

负载均衡：系统外的客户端调用系统内的服务时需要通过反向代理和负载均衡的方式
API网关：系统架构内部服务之间的调用需要通过 API 网关
服务注册与发现：服务之间的互相感知需要用到服务注册与发现
服务通信：服务之间的通信会使用 RPC 架或 HTTP 调用

疑问：Nginx 和 API 网关的区别是什么？

分布式协同

  • 分布式系统的特性和互斥问题（基于以下一系列特性而产生了分布式互斥算法）
    • 特性1：竞争临界资源的进程部署在不同服务器上，进程之间的通信依赖于网络，在请求临界资源时会由于网络原因出现延迟、丢包等情况
    • 特性2：不同服务器上的时间不一定都一致，所以需要全局时钟机制来保证时间的同步
    • 特性3：因为服务在不同机器中，所以需要网络通信，当网络出现故障时，需要一定的检测机制，保证系统整体的稳定性
  • 分布式锁
  • 分布式事务
  • 分布式选举

分布式互斥算法

集中互斥算法
就是在服务之外增加一个全局的协调者，所有服务想要获取或者释放某个资源都需要经过全局协调者

基于许可的互斥算法
不增加额外的协调者，而是内部服务自己决定应该由谁来获的资源，即当某个服务想要获取一个共享资源时，需要获取同类型的所有服务中的半数以上的服务认可才行

分布式事务

事务的 ACID 特性

  • 原子性（Atomicity）：事务是原子的，要么全部成功，要么全部失败
  • 一致性（Consistency）：事务执行前后，数据保持一致性
  • 隔离性（Isolation）：事务执行时，数据保持隔离性，即一个事务的执行不会影响其他事务的执行
  • 持久性（Durability）：事务执行成功后，数据保存在磁盘上

CAP 理论

  • 一致性（Consistency）：所有节点在同一时刻的数据视图是一致的，即对客户端来说，不论访问的是哪台机器，数据都会是最新的
  • 可用性（Availability）：当集群中出现网络分区时，即使当前机器上的数据可能不是最新的，也会响应结果给客户端（响应的数据可能是旧的），一般会通过负载均衡，缓存机制，异步处理，容错机制等方式来提高可用性
  • 分区容错性（Partition tolerance）：分区容错性是指在分布式系统中，即使出现网络分区或者节点故障，系统仍然能够正常运行和提供服务的能力。网络分区是指因为网络故障导致节点之间的通信中断，使得整个系统被分割成若干个孤立的部分。分区容错性要求系统能够在这种情况下，保持数据的一致性和可用性，即对于客户端请求，系统仍然能够返回正确的结果，一般会通过多副本，异步复制、故障转移等策略，将数据在多个节点上进行复制，以确保在某个节点发生故障时，其他节点仍然可以提供服务

为什么 CAP 理论中最多只能满足两个？

  1. 因为分布式系统的特定，分区容错性是一定要满足的，因为分布式系统依赖网络通信，如果因为网络问题，造成数据同步失败，大集群被分割成多个无通信的小集群等，系统应该还是要能提供服务，所以在出现网络问题的情况下，系统还要能提供服务，就代表着一定要满足分区容错性
  2. 在出现网络的问题情况下，若想满足C，即数据必须保持一直，则只能牺牲 A，系统需要等到网络恢复才能用
  3. 在出现网络的问题情况下，若想满足A，则必须能忍受返回的数据不是最新的这种情况，也就是牺牲了 C
